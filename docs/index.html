AI Search Agent

A code-first agentic-framework app that recreates a Microsoft Foundry Azure AI Search agent. It answers questions grounded in Azure AI Search and cites the exact source documents and page numbers — with optional SharePoint content secured by document-level access control.

What is this?

The original agent lives in Microsoft Foundry as a low-code "prompt" agent. This project reproduces it as a maintainable, testable .NET application using the Microsoft Agent Framework, plus the infrastructure to deploy the backing Azure AI Search service and index.

🔎 Grounded answers

Every response is retrieved from the documents-index search index — the agent is instructed never to invent sources.

📄 Page-level citations

Answers include a "Pages Referenced" section listing each source document and the exact page ranges used.

🧩 Reproducible infra

Bicep provisions the search service; a script creates the index schema (fields, semantic & vector config).

✅ Tested

xUnit + Moq cover search options fidelity, the citation payload shape, and prompt drift guards.

🔒 SharePoint ACLs

An optional SharePoint index enforces document-level access control at query time (preview) — users only see what they're permitted to.

Architecture

The user chats with the agent; the model decides when to call the Azure AI Search tool, which returns passages plus document/page metadata used to build cited answers.

Index schema (documents-index)

Key fields that power retrieval and citations:

  • id — key (keyword analyzer)
  • chunk — searchable passage text
  • sourceDoc, title, url — citation metadata
  • pageFrom / pageTo — page range (Int32)
  • vector — 3072-dim, HNSW / cosine (for future vector/hybrid search)

SharePoint with document-level ACLs preview

An optional second index, sharepoint-index, is populated by a SharePoint Online indexer that ingests each item's permissions. At query time the signed-in user's Microsoft Entra token is forwarded, and Azure AI Search trims results to documents that user may access — no custom security-trimming code required.

ACL enforcement relies on the caller's identity, so the app authenticates as the signed-in user (not a service principal). API keys bypass ACLs and are not used for this index.

Quick start

  1. Deploy Azure AI Search 
    az group create --name TRSDemo --location centralus

az deployment group create \
  --resource-group TRSDemo \
  --template-file infra/main.bicep \
  --parameters infra/main.parameters.json
  2. Create the index (data-plane, not part of ARM/Bicep) 
    ./scripts/create-index.ps1 -SearchServiceName trsdemosearch -ResourceGroup TRSDemo
  3. Configure endpoints in src/AiSearchAgent/appsettings.json, or via environment variables: 
    $env:AZURE_OPENAI_ENDPOINT = "https://<your-openai>.openai.azure.com/"
$env:AZURE_SEARCH_ENDPOINT = "https://trsdemosearch.search.windows.net"
  4. Run 
    dotnet run --project src/AiSearchAgent
  5. Test 
    dotnet test
  6. (Optional) Add the SharePoint ACL index — see SharePoint with document-level ACLs. Fill in the data-source connection string, then: 
    ./scripts/create-sharepoint-index.ps1 \
  -SearchServiceName trsdemosearch -ResourceGroup TRSDemo -RunIndexer
    Enable it in appsettings.json under SharePointSearch.
Authentication defaults to DefaultAzureCredential (Entra ID). The identity needs Cognitive Services OpenAI User on the Azure OpenAI resource and Search Index Data Reader on the search service — or provide API keys in config.

How the agent responds

The instructions enforce a consistent, citation-first output format:

  • Begins with a short conceptual checklist (3–7 bullets).
  • Answers only from Azure AI Search results (the documents-index and, when enabled, the ACL-secured sharepoint-index).
  • Ends with a Pages Referenced section grouping page ranges by document, in ascending order.
  • Falls back to "Pages not available" when metadata is missing — and never fabricates sources.
